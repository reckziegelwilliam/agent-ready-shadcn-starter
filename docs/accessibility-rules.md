# Accessibility Rules

This document defines the accessibility standards for all UI code in agent-ready-shadcn-starter, whether written by hand or generated by an AI agent. The baseline is **WCAG 2.1 Level AA**. These rules apply to every component, page, and feature in the repo.

Use this document as a reference during development and as a checklist during review.

## Baseline standard

All UI must meet WCAG 2.1 Level AA. This means:

- All content is perceivable by all users (including those using screen readers or high-contrast modes)
- All functionality is operable via keyboard alone
- Content is understandable (clear labels, predictable behavior, error guidance)
- Content is robust enough to work with current and future assistive technologies

## Keyboard navigation

Every interactive element must be usable without a mouse.

### Requirements

- All buttons, links, inputs, selects, checkboxes, and custom controls must be reachable via Tab key
- Tab order must follow the visual reading order (left-to-right, top-to-bottom)
- Focus must be visible on every focusable element. Use the default shadcn/ui focus ring or the Tailwind utility: `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2`
- Escape key must close any open dialog, dropdown, or popover
- Enter and Space must activate buttons and toggles
- Arrow keys must navigate within composite widgets (tabs, dropdown menus, radio groups)
- No keyboard traps: the user must always be able to Tab away from a component

### Focus management

When the UI changes in response to user action, focus must move to the appropriate element:

| Action | Focus target |
|--------|-------------|
| Dialog opens | First focusable element inside the dialog |
| Dialog closes | The element that triggered the dialog |
| Inline error appears | The first invalid form field |
| Route changes | The main content heading or a skip link target |
| Item deleted from list | The next item in the list, or the previous if it was the last |
| Toast/notification appears | Focus stays where it is (toast is announced via live region) |

### Focus trapping in modals

When a `Dialog` is open:

- Focus must be trapped inside the dialog (Tab should cycle through dialog elements only)
- Focus must not reach elements behind the dialog
- Escape must close the dialog
- When the dialog closes, focus must return to the trigger element

shadcn/ui's `Dialog` component handles this correctly by default. Do not build custom modals that bypass this behavior.

## Screen reader support

### Semantic HTML

Use the correct HTML elements for their purpose:

| Purpose | Element | Not this |
|---------|---------|----------|
| Navigation links | `<a href>` or `<Link>` | `<div onClick>` |
| Actions | `<button>` | `<div onClick>`, `<span onClick>` |
| Page sections | `<main>`, `<nav>`, `<aside>`, `<footer>` | `<div>` |
| Headings | `<h1>` through `<h6>` in order | `<div className="text-2xl font-bold">` |
| Lists | `<ul>`, `<ol>`, `<li>` | Nested `<div>` elements |
| Tables | `<table>`, `<thead>`, `<tbody>`, `<tr>`, `<th>`, `<td>` | CSS grid pretending to be a table |

### Heading hierarchy

- Every page must have exactly one `<h1>`
- Headings must not skip levels (do not jump from `<h1>` to `<h3>`)
- Headings should describe the content that follows them
- Card titles inside a page with an `<h1>` should typically be `<h2>` or `<h3>`

### Live regions for dynamic content

When content changes without a page reload, screen readers need to be notified:

```tsx
// For status messages (polite -- waits for the user to finish current action)
<div role="status" aria-live="polite">
  {statusMessage}
</div>

// For urgent messages like errors (assertive -- interrupts immediately)
<div role="alert" aria-live="assertive">
  {errorMessage}
</div>
```

The `Sonner` toast component in this repo uses `aria-live` regions by default.

### Visually hidden text

When a visual element needs a text label for screen readers but the visible design does not include text, use a visually hidden class:

```tsx
<button>
  <TrashIcon />
  <span className="sr-only">Delete item</span>
</button>
```

Tailwind's `sr-only` class hides text visually while keeping it accessible to screen readers.

## ARIA patterns for common components

### Forms (login, signup, forgot-password)

The auth forms in this repo (`apps/web/components/auth/`) must follow these patterns:

**Labels:**

Every input must have a visible `<Label>` associated with it. The shadcn/ui `Label` component uses `htmlFor` to connect to the input's `id`:

```tsx
<Label htmlFor="email">Email address</Label>
<Input id="email" type="email" aria-describedby="email-error" />
<p id="email-error" role="alert" className="text-sm text-destructive">
  {errors.email?.message}
</p>
```

**Required fields:**

Mark required inputs with `aria-required="true"` or the HTML `required` attribute. Do not rely solely on a visual asterisk.

**Error messages:**

- Connect error messages to their input with `aria-describedby`
- Wrap error messages in `role="alert"` so they are announced when they appear
- Include the field name in the error message: "Email is required" not just "This field is required"

**Form submission:**

- Disable the submit button during submission with `disabled` and `aria-disabled="true"`
- Announce submission status: "Logging in..." or "Account created successfully"

**Complete example:**

```tsx
<form onSubmit={handleSubmit} aria-label="Login form">
  <div className="space-y-4">
    <div className="space-y-2">
      <Label htmlFor="email">Email address</Label>
      <Input
        id="email"
        type="email"
        required
        aria-required="true"
        aria-invalid={!!errors.email}
        aria-describedby={errors.email ? "email-error" : undefined}
      />
      {errors.email && (
        <p id="email-error" role="alert" className="text-sm text-destructive">
          {errors.email.message}
        </p>
      )}
    </div>

    <Button type="submit" disabled={isSubmitting}>
      {isSubmitting ? "Logging in..." : "Log in"}
    </Button>
  </div>
</form>
```

### Data tables (dashboard)

The data table in `apps/web/components/dashboard/` must follow these patterns:

**Table structure:**

- Use semantic `<table>`, `<thead>`, `<tbody>`, `<tr>`, `<th>`, `<td>` elements (shadcn's `Table` component does this)
- Add `<caption>` or `aria-label` to describe the table's purpose
- Mark header cells with `scope="col"` or `scope="row"` as appropriate

**Sortable columns:**

```tsx
<th scope="col">
  <button
    aria-sort={sortDirection === "asc" ? "ascending" : sortDirection === "desc" ? "descending" : "none"}
    onClick={toggleSort}
  >
    Name
    <span className="sr-only">
      {sortDirection === "asc" ? ", sorted ascending" : sortDirection === "desc" ? ", sorted descending" : ""}
    </span>
  </button>
</th>
```

**Row selection:**

- Use `<Checkbox>` with a label for each row: `aria-label="Select row for {item name}"`
- The "select all" checkbox should have `aria-label="Select all rows"`
- Announce selection count changes via a live region

**Pagination:**

The `data-table-pagination.tsx` component should include:
- `aria-label="Pagination"` on the nav element
- Current page announced: `aria-current="page"` on the active page button
- Clear labels on Previous/Next buttons

**Empty and loading states:**

- `data-table-empty.tsx`: Include a descriptive message in the table cell, not just an icon
- `data-table-loading.tsx`: Use `aria-busy="true"` on the table container and `aria-label` on skeleton rows

### Dialogs (modals)

Using the shadcn/ui `Dialog` component:

```tsx
<Dialog>
  <DialogTrigger asChild>
    <Button>Edit profile</Button>
  </DialogTrigger>
  <DialogContent aria-describedby="dialog-description">
    <DialogHeader>
      <DialogTitle>Edit your profile</DialogTitle>
      <DialogDescription id="dialog-description">
        Make changes to your profile information below.
      </DialogDescription>
    </DialogHeader>
    {/* Dialog body */}
  </DialogContent>
</Dialog>
```

Requirements:
- Every dialog must have a `DialogTitle` (even if visually hidden)
- Use `DialogDescription` to provide context for screen readers
- Focus is automatically trapped and restored by the shadcn/ui `Dialog`
- The dialog must be closable via Escape key

### Navigation

**Skip link:**

Add a skip link as the first focusable element on every page:

```tsx
<a
  href="#main-content"
  className="sr-only focus:not-sr-only focus:absolute focus:top-4 focus:left-4 focus:z-50 focus:bg-background focus:px-4 focus:py-2 focus:rounded-md focus:ring-2 focus:ring-ring"
>
  Skip to main content
</a>
```

**Current page indicator:**

Mark the current page in navigation with `aria-current="page"`:

```tsx
<nav aria-label="Main navigation">
  <a href="/dashboard" aria-current={pathname === "/dashboard" ? "page" : undefined}>
    Dashboard
  </a>
</nav>
```

**Dropdown menus:**

The shadcn/ui `DropdownMenu` handles ARIA attributes automatically. Do not build custom dropdown menus with `div` and `onClick`. Use the provided component.

### Tabs

The shadcn/ui `Tabs` component provides correct ARIA roles (`tablist`, `tab`, `tabpanel`) automatically.

When using tabs:
- Each tab panel must have content or `aria-label`
- Selected tab should have `aria-selected="true"` (handled by shadcn/ui)
- Arrow keys should navigate between tabs (handled by shadcn/ui)

## Color contrast

### Minimum ratios

| Content type | Minimum ratio | Example |
|-------------|---------------|---------|
| Normal text (< 18px or < 14px bold) | 4.5:1 | Body text, labels, error messages |
| Large text (>= 18px or >= 14px bold) | 3:1 | Headings, large button text |
| UI components and icons | 3:1 | Input borders, icons that convey meaning |

### How to check

- Use the browser DevTools color picker to check contrast ratios
- Use the Axe browser extension for automated contrast checking
- Test with both light and dark themes

### Common contrast issues

- Muted placeholder text that is too light: use `text-muted-foreground` which is designed to meet contrast requirements
- Disabled elements: while disabled elements are exempt from contrast requirements, ensure they are still distinguishable from the background
- Error states: `text-destructive` must have 4.5:1 contrast against its background in both light and dark mode

### Do not use color alone

Color must not be the only way to convey information:

- **Errors:** Use red text AND an error icon or prefix text like "Error:"
- **Status:** Use a colored badge AND text label (not just a colored dot)
- **Required fields:** Use an asterisk AND `aria-required`, not just red color
- **Links:** Underline links or use another visual indicator beyond color

## Error announcements

Errors must be perceivable by all users, including those using screen readers.

### Form validation errors

When a form is submitted with errors:

1. Move focus to the first invalid field
2. Announce each error with `role="alert"` or `aria-live="assertive"`
3. Connect each error to its field with `aria-describedby`
4. Mark invalid fields with `aria-invalid="true"`

### Network and system errors

When an API call fails or a system error occurs:

1. Display the error in a visible banner or toast
2. Ensure the error container uses `role="alert"` or `aria-live="assertive"`
3. Include actionable guidance: "Could not save. Please try again." not just "Error."
4. If a retry action is available, make the retry button focusable

### Toast notifications

The `Sonner` component provides accessible toast notifications. When using toasts:

- Success messages: Use `role="status"` with `aria-live="polite"` (default behavior)
- Error messages: Use `role="alert"` with `aria-live="assertive"`
- Do not display critical errors only in toasts. Toasts auto-dismiss. Persistent errors need a persistent UI element.

## Reduced motion

Respect the `prefers-reduced-motion` media query:

```css
@media (prefers-reduced-motion: reduce) {
  /* Disable or simplify animations */
}
```

Tailwind provides the `motion-reduce:` variant:

```tsx
<div className="animate-spin motion-reduce:animate-none">
  <Spinner />
</div>
```

All animations and transitions should either respect this preference or be purely decorative and non-disorienting.

## Testing

### Manual testing checklist

Run through this for every new component or page:

1. **Keyboard only:** Unplug your mouse (or do not touch the trackpad). Navigate the entire feature using only Tab, Shift+Tab, Enter, Space, Escape, and arrow keys. Can you reach and operate everything?

2. **Screen reader:** Test with VoiceOver (macOS: Cmd+F5) or NVDA (Windows). Navigate the feature and verify:
   - All interactive elements are announced with their role and label
   - Form fields announce their label, required state, and error messages
   - Dynamic content changes are announced
   - Headings are announced in the correct order

3. **Zoom:** Set browser zoom to 200%. Verify that all content is still visible and usable without horizontal scrolling.

4. **High contrast:** Enable high contrast mode in your OS. Verify that all text and interactive elements remain visible.

5. **Color blindness:** Use a color blindness simulator (browser extension or DevTools). Verify that no information is conveyed by color alone.

### Automated testing

Run these tools as part of the development workflow:

- **eslint-plugin-jsx-a11y:** Catches common accessibility issues at lint time. Already configured in this repo.
- **Axe DevTools:** Browser extension that scans the rendered page for WCAG violations.
- **Lighthouse:** Accessibility audit built into Chrome DevTools (Audits tab).

## Review checklist

Use this checklist when reviewing any UI code, whether written by hand or generated by an AI agent.

### Keyboard

- [ ] All interactive elements are reachable via Tab
- [ ] Tab order matches visual reading order
- [ ] Focus is visible on every focusable element
- [ ] Escape closes dialogs, dropdowns, and popovers
- [ ] Enter/Space activates buttons and toggles
- [ ] Arrow keys navigate composite widgets (tabs, menus, radio groups)
- [ ] No keyboard traps exist
- [ ] Focus moves to the correct element after UI changes (dialog open/close, error, navigation)

### Screen reader

- [ ] Semantic HTML elements are used (button, a, nav, main, h1-h6, table, ul/ol)
- [ ] Every page has exactly one h1
- [ ] Heading levels do not skip
- [ ] All images have alt text (or empty alt for decorative images)
- [ ] All icon-only buttons have sr-only text or aria-label
- [ ] Live regions announce dynamic content changes
- [ ] Tables have captions or aria-labels

### Forms

- [ ] Every input has a visible, associated label
- [ ] Required fields are marked with aria-required
- [ ] Invalid fields are marked with aria-invalid
- [ ] Error messages are connected to fields with aria-describedby
- [ ] Error messages use role="alert" for immediate announcement
- [ ] Submit button is disabled and labeled during submission
- [ ] Focus moves to the first invalid field on submission error

### Color and contrast

- [ ] Normal text has at least 4.5:1 contrast ratio
- [ ] Large text has at least 3:1 contrast ratio
- [ ] UI components (borders, icons) have at least 3:1 contrast ratio
- [ ] No information is conveyed by color alone
- [ ] Design works in both light and dark mode
- [ ] Design is usable in high contrast mode

### Dialogs and overlays

- [ ] Dialog has a DialogTitle (visible or sr-only)
- [ ] Dialog has a DialogDescription or aria-describedby
- [ ] Focus is trapped inside open dialogs
- [ ] Focus returns to trigger element when dialog closes
- [ ] Escape key closes the dialog
- [ ] Background content is inert when dialog is open

### Responsive and motion

- [ ] Content is usable at 200% browser zoom without horizontal scroll
- [ ] Touch targets are at least 44x44px on mobile
- [ ] Animations respect prefers-reduced-motion
- [ ] No content is hidden or inaccessible at any supported breakpoint
